## Leak-fix verification — Basic Ship rebuild soak

The headline of this cycle.  The #274 fix caches
`PlayerShip._extract_ship_texture` by `(faction, ship_type, level)` so
each parked-ship rebuild reuses the **same** `arcade.Texture` object,
instead of minting a fresh one the GPU atlas retained forever.

| Cycle | Mem growth | Result |
|---|---:|:---:|
| 2026-06-16 | +50.0 MB | PASSED |
| 2026-06-18 | +62.0 MB | PASSED |
| 2026-06-20 00:09 (pre-fix) | **+95.6 MB** | **FAILED** (budget 75) |
| **2026-06-20 08:48 (post-fix)** | **+69.0 MB** | **PASSED** |

The fix removed the texture-atlas component of the growth and brought
the soak back under budget (FPS was never the issue — held 267 END /
213 min this cycle).  The per-interval slope is now roughly linear
(+5.6 → +14.4 → … → +59.0 → +69.0) with no texture-driven acceleration.

**Honest caveat (carry-forward WATCH):** +69 MB is back in line with
the pre-leak historical range (+50 / +62), so the texture leak was the
increment that pushed it over — but a **residual ~60–70 MB of
non-texture growth remains**, and that sits close to the 75 MB
threshold.  This is not new (it predates the texture leak) but it
leaves little margin.  Next step if it trends up again: profile the
remaining per-rebuild retention (parked-ship sprite/spatial-hash
churn, station-inventory ops, or GC not reclaiming within the 5-min
window).  Not over budget today; watch-only.
