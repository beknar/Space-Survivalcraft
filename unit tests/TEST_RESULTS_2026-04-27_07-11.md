### Failure 1 (REAL, FIXED): `test_drone_in_entrance_room_with_target_outside_exits`

```
AssertionError: drone failed to exit the maze: still inside
room 0 at (...)
```

**Diagnosis**: regression caused by the drone unified-geometry
zone cache I added in commit `dc5373c`
(`zone._drone_unified_geometry`).  `Drone.attach_maze_planner`
uses `id(rooms) ^ id(room_graph)` as a freshness key — caching
the dicts kept that id stable, so the WaypointPlanner was
constructed ONCE per zone and its internal state (waypoint
history, no-progress timer) persisted across the entire test.
Pre-cache, the geometry dicts were rebuilt every frame, which
forced the planner to be re-created every frame too — masking
this state-retention behaviour.

**Why the cache was added**: I claimed in the original commit
message it would prevent ~25 KB/frame of dict allocation
(~90 MB/min in soak).  That was wrong — Python dicts/lists are
GC'd cheaply and don't accumulate sprite back-references.  The
real soak leak was the Star Maze `_turret_target_list`
SpriteList back-reference accumulation (the *other* fix in
`dc5373c`), which is unaffected by this revert.

**Fix** (commit `da1e07a`): reverted just the drone-cache hunk
in `update_logic.update_drone`.  The Star Maze SpriteList fix
+ the `_turret_extra_target_lists` property + the plain-Python
list for turret targeting — all of which were the actual
soak-leak fix — remain in place.

Verified post-revert:

```
pytest "unit tests/integration/test_drone_pathfinding.py"
15 passed in 18.74s
```

### Failure 2 (FLAKY): `test_drone_does_not_park_on_doorway_midpoint`

Passes in 2 of 3 isolated runs.  Documented in the previous
test report (2026-04-27 earlier in the day) as a back-reference
state-pollution flake driven by the session-scoped `real_window`
fixture combined with the function-scoped `real_game_view`.
Sprite back-references in `sprite_lists` tuples persist across
GameView instances within a single pytest session because
arcade's SpriteLists are kept alive by the sprites that hold
them.  No production-code regression.

**Status**: left as-is; reliably passes in isolation.  Future
work: split flaky drone-pathfinding tests into a separate file
with `@pytest.fixture(scope="function")` for `real_window`
(~50 ms cost per test) — not worth a deeper rework right now.
