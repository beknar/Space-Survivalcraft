## Triage

### Unit suite — 1 pre-existing RSS flake

`test_soak_base_gc.py::TestGetRssMb::test_repeated_calls_return_close_values`
failed (passes in isolation, documented since PR #75).  Same flake
as the prior cycle.  Not a regression in our code.

### Integration suite — PR #157 introduced a memory regression

**The VideoPlayer FPS regression that motivated PR #157 IS fixed**
this cycle:

| Metric | Prior cycle (2026-05-19) | This cycle | Δ |
|---|---:|---:|---:|
| VideoPlayer play/stop churn END FPS | **13.8** | **269.5** | **+255.7 (recovered)** |

But PR #157's switch from `gc.collect()` (full) to `gc.collect(0)`
(gen-0 only) broke the cell-ref cleanup the test soak depends on:

| Metric | Prior cycle | This cycle | Cap |
|---|---:|---:|---:|
| Memory growth at 5 min | ~535 MB | **1576.7 MB** | 1500 MB |

The pyglet audio driver registers closures that capture the
`FFmpegSource` via cell references.  Full `gc.collect()` can break
some of those cycles; gen-0 only catches young garbage, so the
~12 MB/cycle leak documented in the test docstring accumulates ~3×
faster.  Net result: FPS hitching is gone, RSS leak is worse.

**Recommendation:** follow-up PR to revert just the gen-0 change
in `_drain_pending_cleanup` (keep the one-per-call cap).  Or
amortize: full `gc.collect()` periodically (e.g., one full sweep
per ~30 gen-0 sweeps) so the gen-2 cleanup still happens
without the per-frame full-collect cost.

### Basic Ship rebuild — borderline (1.7 MB over)

`test_basic_ship_rebuild_5min_soak`: 76.7 MB grew vs 75 MB cap.
Within typical environmental drift (the cap was already calibrated
in PR #91 to absorb ~5-10 MB run-to-run variance).  Not flagged as a
regression.  If repeats next cycle, candidate for a bump to 80 MB.
