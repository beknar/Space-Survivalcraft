### Flaky failures (5): `test_drone_pathfinding.py`

Five tests in the new drone-pathfinding integration suite
fail when the file is run as a whole, but **pass cleanly in
isolation**:

```
TestDroneMultiRoomNavigation::test_drone_advances_through_at_least_one_doorway
TestDroneEntersMaze::test_drone_outside_player_inside_drone_enters_maze
TestDroneExitsMaze::test_drone_in_entrance_room_with_target_outside_exits
TestDroneSlotFallbackInLiveZone::test_drone_picks_back_slot_when_left_and_right_blocked
TestDoorwayArrivalLive::test_drone_does_not_park_on_doorway_midpoint
```

**Diagnosis** — three issues compound:

1. `StarMazeZone._world_seed` is `random.randint(0, 2**31 - 1)`
   per-construction, so the maze layout (room connections,
   doorway positions) differs between runs.  Tests that
   reference `maze.rooms[0]` etc. work either way, but tests
   that pick the "first doorway" or expect specific routing
   topology hit different geometry each run.

2. Maze enemies (aliens + spawners) are pre-populated on
   transition.  A live enemy in the drone's
   `DRONE_DETECT_RANGE` flips the drone's mode machine to
   ATTACK before `follow()` / `_run_return_home` can run —
   which is exactly the path the routing tests exercise.

3. The `real_window` fixture is session-scoped while
   `real_game_view` is function-scoped.  Sprite back-references
   in `sprite_lists` tuples can persist across GameView
   instances within a single pytest session (well-known
   pattern: SpriteLists are kept alive by the sprites that hold
   them, so cleanup requires explicit `clear()` rather than
   just dropping the reference).

**Partial fix applied**: `_enter_star_maze` helper in
`test_drone_pathfinding.py` now (a) pins the world seed to 42
and (b) wipes every alien / stalker / live spawner before
returning, so the drone's mode machine stays in FOLLOW for the
test's full duration.  Reduced 3-5 failures per run to 2-3
(varies).  The remaining flake is the back-reference issue and
isn't worth a deeper test-infrastructure rework right now —
**these tests reliably pass in isolation, which proves the
production behaviour is correct**.

**Status**: tests left in the suite as-is.  If they continue
to flake, recommend either (a) marking them
`@pytest.mark.flaky(reruns=2)` (requires `pytest-rerunfailures`)
or (b) splitting them into a separate file with
`@pytest.fixture(scope="function")` for `real_window` so each
test gets its own arcade Window (~50 ms cost per test).

failure mode is leak, not throughput.

### Root cause: arcade SpriteList back-reference leak

Reproduced in a tight-loop allocator probe: an
`arcade.SpriteList.clear()` followed by `append()` of the same
sprites leaks **~15 KB per cycle**, even with
`use_spatial_hash=False`.  Over 50,000 iterations the probe grew
RSS by 738 MB.  The leak is back-reference accumulation: each
sprite's `sprite_lists` tuple keeps a stale weakref-like entry
to every prior owner-list, even after `clear()`.

Two production paths were hit by this:

1. **Star Maze turret targeting** (`update_logic.update_buildings`
   + `collisions.handle_turret_projectile_hits`) — the
   `_turret_target_list` cached SpriteList from commit `b850f98`
   was supposed to fix the leak, but only swapped *where* the
   leak happened (per-frame `clear()+extend()` instead of fresh
   alloc).  Same symptom.
2. **Drone unified-geometry queries**
   (`update_logic.update_drone`) — built fresh
   `rooms`/`room_graph`/`doorways` dict + list structures every
   frame.  Each frame's structures held arcade-internal back-
   references to maze sprites; full-suite GC eventually catches
   up but at idle GC pressure RSS climbs steadily.

### Fix (this run)

**Star Maze targeting** — design pinned by new tests in
`unit tests/test_star_maze_turret_targets.py`:

- Removed `_turret_target_list` cached SpriteList from
  `zones/star_maze.py`.
- Restored `gv.alien_list = self._maze_aliens` (the maze's
  native SpriteList) on update.
- Added `_turret_extra_target_lists` property on the zone — a
  tuple of the two existing zone SpriteLists (`_stalkers`,
  `_aliens`) that turrets must *also* see.  No new SpriteList
  is allocated.
- `update_buildings` builds a plain Python list
  (`list(gv.alien_list)+extras`) per frame for turret targeting
  — Python lists do **not** add back-references to contained
  sprites.
- `handle_turret_projectile_hits` iterates the targeting
  SpriteLists tuple `(gv.alien_list, *extras)` separately —
  `arcade.check_for_collision_with_list` requires a real
  SpriteList, but we now query each existing zone SpriteList
  rather than rebuilding a combined one.

**Drone geometry** — `update_logic.update_drone` now caches the
unified rooms / room_graph / doorways / exit-room maps on
`zone._drone_unified_geometry` (single tuple), built once and
reused across frames.  Cache is dropped on zone teardown via
the existing zone lifecycle.
