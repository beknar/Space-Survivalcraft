Example from this run:

```
[Video player] START: 3.6 FPS, 237 MB RSS

[Inventory churn] START: 3.3 FPS, 244 MB RSS

```

Performance tests on the same HEAD in the same session ran at 100-500+ FPS. Fast suite ran in 5.45 s. The host was doing *something* (likely background indexing / thermal throttling) during the soak attempt. This matches the pattern: **perf tests fresh-GameView-60-frames stay healthy; soak tests that run for 5 minutes each collapse uniformly**. Not a code regression — same outcome as the prior three soak attempts.

The run was stopped after 50 min / 8 scenarios to free the host for a targeted Star Maze soak run.

### Star Maze soak — isolated run (3 / 3 FAILED, but for real reasons)

Ran `pytest "unit tests/integration/test_soak_star_maze.py"` on its own. **All three scenarios ran at healthy FPS** — host contention had cleared. This made the run informative, and it surfaced a **real memory leak in the Star Maze zone** that the FPS-only soak-triage wouldn't have caught.

| 4.0 m | 112.2 | 654 MB | +447.3 MB |
| 4.5 m | 122.3 | 710 MB | +503.1 MB |
| END   | 125.2 | 760 MB | **+553 MB** |

FPS: **72-141 FPS steady-state** (threshold 40). Memory: **+553 MB** over 5 minutes (threshold **+50 MB**). Assertion: `mem_growth <= max_memory_growth_mb`.

**Diagnosis:** ~110 MB/min growth while idle. No combat, no pickups, just the zone ticking. The player is far from every maze so no alien pursues, but the four spawners continue their 30 s spawn cadence (though alive_children is already at cap so no new aliens queue). Suspects to investigate:

- `_walls_near` returns a fresh `list[Rect]` + `set[int]` every call — maybe 2-3 K calls/frame.
- `_advance_alien_projectiles` allocates a new `prevs: list` every tick.
- Arcade SpriteList texture-atlas rebuilds when sprites enter/leave the list (spawners rotate their hit-tint).
- The fog grid rebuild / minimap texture refresh.

All of these are small per-tick but at 60 FPS × 300 s they compound. The leak is not in the maze geometry itself (which is static).
