### `test_drag_zorder_and_starmaze_ai.py::TestStarMazeAIPilotPatrol::test_ai_pilot_actually_moves_in_star_maze`

The test plants a Home Station + a parked ship with `ai_pilot`
installed in the Star Maze and asserts the ship moves > 100 px in
1 second of simulated time.  Pre-fix failure rate: 4 / 5 runs in
isolation (~80 %).  The flake had been intermittently appearing
since at least PR #33's triage cycle but was attributed to other
causes — this cycle pinned the actual root cause.

**Root cause**: `StarMazeZone._world_seed` is initialised via
`random.randint(0, 2**31 - 1)` at zone construction, so each test
run gets different Z2 alien spawn positions inside the maze.  When
an alien randomly lands within `AI_PILOT_DETECT_RANGE` (600 px) of
the parked ship at (6100, 6000), the AI pilot enters **engage
mode** (lateral chase at AI_PILOT_SPEED = 140 px/s) instead of
**patrol mode** (snap-to-orbit-ring producing ~260 px of
displacement on the first frame).  Engage motion in 1 s is well
under the 100 px threshold the test asserts.

**Fix**: in the test's `_setup`, clear every enemy list on the zone
(`_aliens`, `_maze_aliens`, `_stalkers`) plus `gv.alien_list` after
`_transition_zone` so the patrol code path is isolated.  Engage
mechanics are covered by the dedicated AI-pilot combat suite —
this test specifically exercises patrol.  Verified 10 / 10 passes
after the fix; previously 4 / 5 failed.

No code change.  This is a stale test isolation that wasn't
catching the random-seed coupling.

## Notes for next cycle

- **A* live-validation** is the next natural step — the unit /
  integration tests cover the planner's algorithm and dock-radius
  relaxation, but a real bot session against the live game has
  not yet exercised the planner end-to-end.
- **Stuck-detect 4-gate model** — worth a follow-up doc summary.
  Each gate has its own constants (`STUCK_DETECT_*`,
  `WALL_PIN_TRAP_*`, `MINE_NO_PROGRESS_S`, the new centroid-shift
  gate from #74); contributors landing future gates have to read
  4 PR descriptions to understand the layering.
- **Test-isolation pattern**: the flake fix in this cycle exposed
  a class of latent fragility where tests rely on
  `random.randint`-based world seeds for deterministic behaviour.
  The standard-library `random.seed` doesn't get a per-test
  reset, and zone construction calls it for the world seed.  A
  conftest-level `monkeypatch.setattr` of the zone's seed
  generator (or a fixture that pins seeds in tests that depend on
  them) would canonise the fix.  Worth a follow-up.
- **Coverage churn**: this cycle added more tests than several
  prior cycles combined (+73 net) but most are concentrated in
  the bot-autopilot helpers.  The docs-only PRs (#68, #72)
  contribute nothing to coverage, which is fine but worth noting
  for cycle-summary trend graphs.
